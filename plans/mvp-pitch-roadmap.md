# MVP Accord — план к питчу со Сбером

## Зафиксированные решения

- **Backend**: Supabase (Postgres + Auth + Storage + Edge Functions)
- **OTP-провайдер**: SMS.ru (Twilio не используем — нестабильная доставка на RU-номера)
- **Email**: Resend (отправка чеков/саммари)
- **Push**: Web Push (браузер) + SMS-уведомление на телефон как доп. канал
- **Карты**: 2 статичные виртуальные карты-костыль (реальные карты пользователя), показываются в открытом виде всем пользователям без шифрования (осознанный риск на этапе теста):
  - МИР: 2202 2082 8064 0658, 09/35, CVC 914 (для платежей внутри страны)
  - MC: 5269 8800 1729 4802, 01/31, CVC 624 (для зарубежных, рекомендован перевод с Альфа-Банка — 0% комиссия)
- **Пополнение баланса**: ручное подтверждение через Supabase Table Editor / админку (авто-парсинг банковских SMS технически невозможен в web — требует Android-приложения с разрешением на чтение SMS, чего нет в вебе и недоступно на iOS). Архитектура БД закладывается так, чтобы потом заменить ручное подтверждение на вебхук/парсер без изменения схемы.
- **Подтверждение привязки карты к сервису**: пользователь жмёт "Я привязал" (статус `user_confirmed`), система отдельно помечает `system_confirmed` после первого реального списания на эту подписку.
- **Реестр подписок**: своя таблица популярных RU-сервисов (лого, тарифы) + AI-поиск как fallback + конструктор кастомной подписки как последний fallback.
- **Язык**: всегда `ru`. Переключатель скрыт в ProfileTab. EN-переводы и Global-банки (Revolut/Mercury) физически не удаляются, просто недостижимы через UI.
- **BNPL**: модалка и кнопка "Оформить отсрочку" полностью скрыты из UI (код остаётся).
- **Главный экран**: заменяется на экран регистрации (телефон → SMS-код через SMS.ru).
- **Профиль**: попап "Закончите заполнять профиль" (фото, имя, фамилия, email) всплывает от иконки профиля, когда пользователь привязал все карты ко всем сервисам.
- **Единая дата/сумма**: дата — один день месяца, к которому на карте должны быть деньги; сумма — авто-расчёт суммы всех активных подписок (не редактируется). Сервисы продолжают списывать в свои обычные даты — Accord не меняет даты у самих сервисов.

## Архитектура БД (Supabase/Postgres) — черновая схема

- `users` (Supabase Auth + profile: phone, name, surname, email, avatar_url, profile_completed_at)
- `cards` (id, user_id, type: mir/mc, masked, статичные данные на старте)
- `registry_services` (id, name, logo_url, default_price, category)
- `subscriptions` (id, user_id, registry_service_id nullable, name, price, logo, period, binding_status: pending_user_confirm/user_confirmed/system_confirmed, card_id)
- `charges` (id, subscription_id, due_date, amount, status: pending/due/success/failed, confirmed_by: admin/system)
- `balance_topups` (id, user_id, amount, method, confirmed_by: user/admin, status)
- `user_settings` (user_id, payment_day, computed_monthly_total)

## Фазы реализации

### Фаза 0 — Backend foundation
- Создать проект Supabase, схему БД выше, настроить Auth (Phone OTP via SMS.ru), Storage bucket для фото профиля.

### Фаза 1 — Frontend quick wins (язык, BNPL)
- Залочить язык на 'ru', скрыть переключатель в ProfileTab
- Скрыть кнопку BNPL + не рендерить BNPLModal

### Фаза 2 — Регистрация / Авторизация
- Экран телефон → SMS код (SMS.ru + Supabase Auth)
- Редирект: новый юзер → онбординг, существующий → Home

### Фаза 3 — Профиль
- Экран профиля (фото, имя, фамилия, email)
- Попап "Закончите профиль", триггер — после привязки всех карт ко всем сервисам

### Фаза 4 — Выпуск карт МИР/МС
- Экран выбора карты(карт), отображение статичных номеров

### Фаза 5 — Реестр + поиск подписок
- Таблица registry_services + стартовый сид топ-RU-подписок
- Поиск по реестру → AI fallback → конструктор кастомной подписки

### Фаза 6 — Экран выбора найденных подписок
- Список с чекбоксами, редактируемый тариф, добавление выбранных

### Фаза 7 — Единая дата и сумма
- Выбор дня месяца + авто-расчёт суммы

### Фаза 8 — Пополнение карты (инструкция)
- Экран с номерами карт и инструкцией, кнопка "Я перевёл", ручное подтверждение баланса админом

### Фаза 9 — Привязка карты в сервисах
- Экран-инструкция на каждую подписку + кнопка "Я привязал"

### Фаза 10 — Напоминания
- Web Push + SMS за 1-2 дня до даты списания

### Фаза 11 — Оркестрация списаний (ручная) + чек на email
- Таблица charges, ручное подтверждение, Resend-отправка чека

## Открытые вопросы (требуют доработки позже)

- **Выпуск второй карты позже**: сейчас `CardIssueScreen` показывается только один раз — сразу после первого входа, пока у пользователя 0 карт. Не продумано, где и как пользователь, выпустивший только МИР, сможет позже довыпустить MC (или наоборот). Вероятно, нужна отдельная точка входа в Профиле ("Мои карты" → "Выпустить ещё карту"), которая открывает тот же `CardIssueScreen`, но в режиме "докупки" (показывая только невыпущенные типы).
- **AI-поиск как fallback**: пока не подключён. Если сервис не найден в реестре (`registry_services`), сразу предлагается ручной конструктор (`SubscriptionModal`), минуя AI-этап из исходного плана. Нужно решить, подключать ли реальный AI/веб-поиск (требует API-ключ и отдельную интеграцию) или этот шаг не критичен для пилота.
